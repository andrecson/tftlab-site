# src/server/ — server-only modules

- `db.ts` — the PrismaClient singleton. Import everywhere as `import { db } from "@/server/db"` (cached on globalThis in non-production).

## ddragon.ts — TFT catalog loader (US-009)
- **Source is Community Dragon**, not classic Data Dragon. Riot's `ddragon.leagueoflegends.com` TFT files only expose id/name/cost/image — they carry NO trait breakpoints and NO champion↔trait associations, both of which the synergy engine + builder require. `getCatalog()` reads the full export from `raw.communitydragon.org/latest/cdragon/tft/en_us.json`. The file/function keep the `ddragon` name per the PRD.
- **Current set is resolved dynamically**: highest-numbered mainline set (mutator `TFTSet<number>`, no suffix). The returned `set` token (e.g. `"TFTSet17"`) is the canonical value to store in the `set` column across all models. No hardcoded set number — it follows `latest`.
- **Icons** are `.tex` asset paths rewritten to servable CDN PNGs by `assetUrl()`: `raw.communitydragon.org/latest/game/<lowercased-path>.png`. Champion icon uses `tileIcon || squareIcon || icon`. These URLs resolve (HTTP 200 image/png) — US-042 must add `raw.communitydragon.org` to `next.config` `images.remotePatterns`.
- `getCatalog()` returns normalized `champions/traits/items/augments` already filtered, classified, and de-duped by apiId; it performs NO DB writes (pure transform of remote data). Items are classified into `ItemType` by apiName keyword + composition; augments are scoped to the current set token only. Trait `breakpoints` are sorted distinct `minUnits` (a couple of meta-traits like "Choose Trait" legitimately have `[]`).

## queries/ — public read queries (US-012)
- Public pages read comps through `src/server/queries/*`, NOT ad-hoc Prisma calls. Three modules: `config.ts` (`getSiteConfig()` → SiteConfig+currentPatch, `getCurrentSet()`), `tierlist.ts` (`getPublishedComps(set?)`, `groupCompsByTier`, `TIER_ORDER`), `comp.ts` (`getCompBySlug(slug)`).
- **Every public query filters `status: "PUBLISHED"`** — drafts/archived must never reach public pages. `getCompBySlug` uses `findFirst` (slug + status) not `findUnique` (findUnique's where only takes unique fields) and returns `null` for missing/draft/archived so the page can 404.
- **`set` defaults to `SiteConfig.currentSet`** when the arg is omitted (`getPublishedComps` → `getCurrentSet()`), returning `[]` if unconfigured. Pass an explicit set only when you need a non-current one.
- `comp.ts` also exports `getPublishedCompSlugs()` (US-017) → all PUBLISHED comp slugs (NOT set-scoped) for the comp route's `generateStaticParams`. Every published comp gets a static page regardless of set. It further exports `getPublishedCompsForSitemap()` (US-023) → `{ slug, updatedAt }[]` (PUBLISHED only) consumed by `app/sitemap.ts` (`updatedAt` → `<lastmod>`). Both are the ONLY things that should enumerate published comps for build/SEO — don't re-query `db.comp` ad-hoc in a route file.
- **Include shapes are `Prisma.validator<Prisma.CompInclude>()({...})`** so the derived row type stays exact; export the row type via `Prisma.CompGetPayload<{ include: typeof theInclude }>` (e.g. `CompCard`, `CompDetail`) for UI components to import. `satisfies`/plain objects widen `orderBy: "asc"` to `string` and break `CompGetPayload` — use `Prisma.validator`.
- Card query (`CompCard`) is lean: main traits (ordered) + carries only (`units: { where: { isCarry: true } }`). Comp scalars (incl. `patchIntroducedId`/`patchUpdatedId`) come for free — pass a `CompCard`/`CompDetail` straight into `getCompBadges(comp, SiteConfig.currentPatchId)`.
- Detail query (`CompDetail`) includes the full tree: traits, `units{ champion, items{ item } }` (all ordered by `order`), `itemPriority{ item }`, `augments{ augment }`, and both `patchIntroduced`/`patchUpdated`. Board positions are `boardRow`/`boardCol` scalars on each unit (null = off-board EARLY/FLEX). The `compDetailInclude` shape is **exported** (US-040) so non-public callers can reuse the exact same `CompDetail` type + the `comp-*` section components — `admin.ts`'s `getAdminCompForPreview(id)` (draft preview) does `db.comp.findUnique({ where:{id}, include: compDetailInclude })` (the ONLY difference from `getCompBySlug` is dropping the `status:"PUBLISHED"` filter so DRAFT/ARCHIVED comps resolve). If you need to render a comp exactly like the public page from anywhere else, reuse this include — don't build a new one.
- **Tier grouping/presentation moved to `src/lib/tiers.ts`** (US-016). `tierlist.ts` re-exports `TIER_ORDER` and `groupCompsByTier` now delegates to `groupByTier` there. This is because `"use client"` components must NOT import this module (it pulls in `db`/Prisma) — the client-safe pure helpers (`TIER_ORDER`, `groupByTier`, `TIER_META`) live in `src/lib/tiers.ts` instead, and the server keeps a single source by re-exporting. When adding new pure list helpers a client component also needs, put them in `src/lib/*` and re-export here, don't duplicate.
- `catalog.ts` (US-025) exposes the champion palette for the builder: `getBuilderChampions(set?)` → `BuilderChampion[]` (`{ id, apiId, name, cost, iconUrl, traits: {id,name,iconUrl}[] }`), set defaults to `getCurrentSet()`, ordered by cost then name, traits sorted by name. It returns PLAIN serializable objects (a mapped shape, NOT Prisma model instances, no Date fields) so a `"use client"` component can receive it as props and it survives `unstable_cache` (JSON) without the Date-string gotcha. Champion traits here are all traits (the schema has no origin/class flag) — the builder palette approximates "origem" = first trait, "classe" = last trait. US-026 added the sibling `getBuilderTraits(set?)` → `BuilderTraitInfo[]` (`{ id, name, iconUrl, breakpoints: number[] }`, `Trait.breakpoints` Json coerced via `toBreakpoints` — array of positive numbers, else `[]`) for the live synergy panel, keyed by the SAME trait `id` this query puts on each champion trait. The builder maps a `BuilderTraitInfo` → `TraitInfo` as `{ key: id, name, breakpoints }` for `computeSynergies`. US-027 added two more siblings for the builder item panel + augment picker: `getBuilderItems(set?)` → `BuilderItem[]` (`{ id, apiId, name, iconUrl, type: ItemType }`, ordered by name — `type` is the raw `ItemType` enum that drives the panel tabs: craftables=COMPLETED, radiants=RADIANT, artifacts=ARTIFACT, componentes=COMPONENT, outros=EMBLEM|SUPPORT|OTHER) and `getBuilderAugments(set?)` → `BuilderAugment[]` (`{ id, apiId, name, iconUrl, tier: string|null }`, ordered by name). Both default to `getCurrentSet()`, return `[]` when unset, and return PLAIN mapped objects (no Prisma instances / Dates) so they survive the JSON cache and pass to the `"use client"` builder as props. The builder page's `["builder-catalog"]` `unstable_cache` entry now bundles all FOUR: `{ champions, traits, items, augments }` (all plain, no Date). All four use the SAME `id` key: champion/trait ids feed the synergy panel; item ids are stored in `PlacedUnit.items` and augment ids in the builder's board-level augment selection (US-028 encodes both into the share code).
- `builder-catalog.ts` (US-028) exports `getBuilderCatalog()` — the single `unstable_cache(["builder-catalog"], { tags: ["catalog"] })` bundle `{ champions, traits, items, augments }` (all four `getBuilder*` queries). BOTH builder routes import it (`/builder` and the shared `/builder/[code]`) so there's ONE cache entry/source. When a route needs the builder catalog, import `getBuilderCatalog` — don't re-wrap the four queries in a new `unstable_cache` (that creates a second, divergent cache key). The shared route additionally sanitizes a decoded share code against this catalog (drops champion/item/augment ids not present) so a stale/cross-set code renders no phantom units.
- `admin.ts` (US-031) exposes `getAdminDashboardStats()` → `AdminDashboardStats` (`{ draftCount, publishedCount, archivedCount, currentSet, currentPatch: { version, releasedAt } | null }`). Admin reads are the ANTI-pattern of the public queries: **NOT** wrapped in `unstable_cache` — the admin shell is `force-dynamic`, so counts must be live (a curator publishing/archiving needs the numbers to move immediately). Counts are set-scoped to `SiteConfig.currentSet` (falling back to all-sets when unconfigured) via a `Promise.all` of three `db.comp.count`s. Admin dashboard/list pages should read stats through this module, not ad-hoc `db.comp.count` calls. US-033 added `getAdminComps(status?)` → `AdminCompRow[]` (lean `Prisma.validator` `select`: `id/slug/name/tier/status/situational/updatedAt` + `patchIntroduced/patchUpdated.version`), same set-scope + not-cached (force-dynamic) rules, `orderBy updatedAt desc`; the `status` filter maps the dashboard's `?status=draft|published|archived` query token → `CompStatus` enum. The comps list lives at `src/app/admin/(admin)/comps/page.tsx` (dashboard StatCards + nav already link there); it filters via `searchParams.status` (a `Promise` in Next 15 — await it) and each row's `Editar` links to `/admin/comps/[id]` + a `Nova comp` link to `/admin/comps/new`. US-034 added `getAdminCompById(id)` → `AdminCompEdit | null` (base + guide fields for the edit form: `id/slug/name/set/tier/situational/status/playstyle/difficulty/whenToPlay/earlyGame/midGame/lateGame/tips/patchIntroducedId/patchUpdatedId`; NOT set-scoped — edit by id; traits/units/items/augments excluded, they're US-035..037) and `getPatches()` → `PatchOption[]` (`{ id, version }`, newest first) for the form's introduced/updated selects. The form routes are `src/app/admin/(admin)/comps/new/page.tsx` + `.../comps/[id]/page.tsx` (the static `new` segment wins over dynamic `[id]`, so `/admin/comps/new` never resolves as id="new"); both are under `(admin)` so they inherit the force-dynamic guard/shell, and both server-render the shared `"use client"` `<CompForm patches comp?>` (`src/components/admin/comp-form.tsx`). Mutations live in `src/actions/comps.ts` (see that dir's CLAUDE.md). US-035 added `getAdminCompComposition(id)` → `AdminCompComposition` (`{ traits: {traitId,level}[], earlyUnitIds: string[], flexUnitIds: string[] }`) — the comp's `CompTrait`s (ordered) plus its off-board EARLY/FLEX `CompUnit` champion ids per role (CORE excluded — the builder owns the board, US-037); same not-cached/not-set-scoped rules. The edit page (`.../comps/[id]/page.tsx`) now also fetches `getBuilderChampions(comp.set)` + `getBuilderTraits(comp.set)` (scoped to the comp's OWN set, not the current set) and renders the `"use client"` `<CompCompositionForm compId champions traits initial>` (`src/components/admin/comp-composition-form.tsx`) BELOW `<CompForm>`. That form reuses `<CatalogPicker>`: a SINGLE-mode picker with `value={null}` as a pure "add" control (options pre-filtered to exclude already-added ids) for traits (added traits render as an ordered list with a level `<input>` + up/down reorder), and MULTIPLE-mode pickers for the EARLY/FLEX unit lists (selection order = list order). It only appears in EDIT mode (a comp needs an id first); the `new` page redirects to `[id]` after create so the curator can then fill traits/units. US-036 added `getAdminCompPriority(id)` → `AdminCompPriority` (`{ itemIds: string[], augmentPriority: AugmentCategory[] }`) — the comp's `CompItemPriority` item ids (ordered) plus the scalar `Comp.augmentPriority` array; same not-cached/not-set-scoped rules, returns empty lists for a missing comp. The edit page also fetches `getBuilderItems(comp.set)` and renders the `"use client"` `<CompPriorityForm compId items initial>` (`src/components/admin/comp-priority-form.tsx`) BELOW `<CompCompositionForm>`. That form reuses the add-picker + ordered-list-with-reorder pattern from the composition editor: a SINGLE-mode item `<CatalogPicker>` (`value={null}`, options exclude already-chosen) feeds an ordered `<ol>` (up/down/remove), and the augment categories are an ordered `<ol>` (up/down/remove) plus "+ Economia/Itens/Combate" add buttons for the not-yet-chosen categories. Mutations are in `src/actions/comps.ts` (`updateCompPriority`). US-037 added `getAdminCompBoard(id)` → `AdminCompBoard` (`{ units: { championId, row, col, stars, isCarry, items: string[] }[], augmentIds: string[] }`) — the comp's on-board **CORE** `CompUnit`s (only `role: CORE` with a non-null `boardRow`/`boardCol`, ordered; `stars = starLevel ?? 1`, items ordered) plus its recommended `CompAugment`s.
